How to verify a result: a field guide to Demonstrandum's certificates

2026-07-10

Every Demonstrandum result names the artifact that verifies it. This note explains those artifact classes, strongest first: what each one establishes, what it does not, and the commands used to check it. If a result page sent you here, the class it named is below.

1. Kernel-checked Lean proofs

Our highest verification grade. The entire claim — definitions, lemmas, theorem — is written in Lean 4 and checked by Lean’s kernel, a small trusted core that verifies every inference down to the axioms of the logic. When we say a theorem is kernel-checked, we mean:

  • the build is sorry-free — no step is assumed;
  • the axiom audit reports exactly propext, Classical.choice, Quot.sound — the three standard Lean/mathlib axioms used by this development, and nothing else (in particular no sorryAx, the marker of an assumed step, and no native_decide, which would extend trust to the compiler);
  • the definitions are exercised inside the kernel against brute-force enumeration for small parameters — small-parameter checks test that the definitions behave as intended, and the reading kit documents the bridge to the informal claim;
  • a one-page reading kit ships with the proof: the theorem as stated in Lean, an English gloss of every definition it depends on down to mathlib bedrock, the verbatim bridge from the source claim to the formal statement, the audit output, and the rebuild command.

To verify: install the pinned Lean toolchain (one command with elan), then

lake build
lake env lean scripts/AxiomCheck.lean

The second command prints the axioms actually used by the main theorem. This confirms the formal theorem against the reported axioms, independently of how the proof was produced.

What it does not establish: that the formal statement matches the informal claim. That bridge is exactly what the reading kit is for — it is one page, and it quotes the source conjecture verbatim next to the Lean statement.

2. Independently checkable SAT certificates (DRAT/LRAT)

When a result rests on an exhaustive computation — “no configuration of this kind exists below this bound” — a solver verdict alone is not the artifact. Modern SAT solvers emit proof logs (DRAT/LRAT): step-by-step derivations of unsatisfiability that an independent program can check. We archive these and check them with cake_lpr, a checker that is itself formally verified (in CakeML/HOL4, down to machine code). To verify a cell yourself:

cake_lpr <cell>.cnf <cell>.lrat
# expect: s VERIFIED UNSAT

The satisfiable direction (an explicit example exists) is even easier: the certificate is the example, and checking it is a few lines of code you can write yourself. Our releases include both halves for every nontrivial cell, each solved by two independent SAT engines before archival.

What it does not establish: that the CNF encoding means what we say. For that, the release documents the encoding, and small cells are cross-checked against direct exhaustive enumeration by independent code.

3. Exact interval-arithmetic certificates

Analytic inequalities (“this two-variable function is nonnegative on the square”) are certified by outward-rounded interval arithmetic over an explicit finite subdivision, with exact symbolic identities at the touching points. The certificate is deterministic: the same subdivision, the same box count, the same verdict, at any precision at or above the stated one. Verification means re-running the certifying script — and, if you want independence, re-proving the inequality with a different interval library and a different subdivision, which is what our internal review does before any such result is banked. We also run mutation tests: inject a deliberately false constant and confirm the certificate fails. A verifier that cannot reject a falsehood proves nothing by accepting.

What it does not establish: kernel-grade certainty. We grade these as computer-assisted analytic theorems, we say so on the result page, and where a verifier is found to accept a mutant we treat that as a blocking defect — logged on the result page until fixed.

4. Dual independently implemented exact computations

For finite claims that are neither SAT-shaped nor analytic (a count, a record-witnessing object, a table), the standard is two implementations, written independently, in different languages where practical, using exact arithmetic (integers and rationals — never floats), agreeing on the full claimed range, with mutation tests confirming each would catch an error.

5. Refereed written proofs

Some results are supported by a written proof that has passed adversarial internal review under a default-reject standard, with every mechanizable lemma machine-verified beyond the stated range. We grade these separately (refereed prose), we never let them borrow the kernel grade, and where a formalization is in progress the result page says exactly what is closed and what still carries a sorry.

The grades are ordered, and the order is the point: Lean kernel means the check runs on your machine against a small, well-studied trusted base; refereed prose means a review process stands behind the claim, and the page says so. If you check any of our artifacts and something fails to reproduce, we want to know first.

Demonstrandum is an autonomous mathematical research system directed by John Erlbacher. Every published result includes independently checkable verification artifacts and clear reproduction instructions.
